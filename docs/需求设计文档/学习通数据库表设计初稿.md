## 2. 设计原则

### 2.1 技术建议

| 项 | 建议值 | 说明 |
| --- | --- | --- |
| 数据库 | MySQL 8.0 | 兼容 JSON、索引和常规后台管理场景 |
| 存储引擎 | InnoDB | 支持事务与行级锁 |
| 字符集 | `utf8mb4` | 支持完整多语言字符 |
| 主键类型 | `bigint` | 满足后期数据增长 |
| 时间字段 | `datetime` | 统一记录业务时间 |
| 命名规范 | `snake_case` | 表名、字段名统一小写下划线 |

### 2.2 建模原则

1. 前台用户与后台管理员共用用户主表，通过角色关系区分权限。
2. 常用业务主表采用逻辑删除字段 `is_deleted`，日志与统计表采用物理保留。
3. 列表查询常用条件如状态、时间、关键字、外键均应建立索引。
4. 数量类字段如点赞数、评论数、收藏数允许冗余存储，以提升前台读取性能。
5. 统计模块优先由业务表汇总生成，若数据规模变大，可落地为日统计汇总表。
6. 题库、练习、消息模板、搜索运营、举报治理等拓展能力在数据库层提前预留主表，避免后续大改结构。
7. 需要灵活扩展的多值字段可优先使用 JSON 或快照字段，但关键检索条件仍应拆分为结构化字段。

### 2.3 公共字段建议

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |
| `created_by` | bigint | 创建人，可空 |
| `updated_by` | bigint | 更新人，可空 |
| `is_deleted` | tinyint | 逻辑删除标记：0-未删除，1-已删除 |
| `remark` | varchar(255) | 备注信息，可空 |

## 3. 核心实体关系概览

| 模块 | 核心表 | 关系说明 |
| --- | --- | --- |
| 用户与权限 | `sys_user`、`sys_role`、`sys_permission`、`sys_user_role`、`sys_role_permission` | 用户与角色、角色与权限均为多对多 |

| 用户互动 | `user_favorite`、`user_follow_source`、`garden_contact_message` | 用户收藏、关注、留言与主业务关联 |

| 学习与练习 | `user_study_record`、`user_offline_download`、`practice_question_bank`、`practice_question`、`practice_paper`、`practice_paper_question`、`user_practice_record`、`user_practice_answer`、`user_wrong_question` | 学习记录、题库、试卷、答题记录形成学练闭环 |

| 消息客服 | `sys_message`、`sys_message_receiver`、`service_ticket`、`service_message`、`service_faq` | 系统消息与用户阅读状态分离，客服按会话和 FAQ 管理 |
| 风控治理 | `risk_report_record`、`sys_sensitive_word` | 举报处理、敏感词规则支撑内容治理 |
| 系统配置 | `sys_config`、`sys_feature_switch` | 系统文本配置与开关配置分离 |
| 日志统计 | `sys_operation_log`、`sys_login_log`、`stat_user_daily`、`stat_course_daily`、`stat_active_daily` | 审计日志与日统计汇总独立 |

## 5. 表关系说明

### 5.1 用户与权限关系

1. `sys_user` 与 `sys_role` 通过 `sys_user_role` 建立多对多关系。
2. `sys_role` 与 `sys_permission` 通过 `sys_role_permission` 建立多对多关系。
3. 后台管理员、超级管理员和前台用户统一存于 `sys_user`，通过角色区分使用范围。
4. `auth_session.user_id`、`auth_verify_code.target_value` 分别支撑登录态管理和找回密码验证。

### 5.2 课程与学习关系

1. `edu_category` 通过 `parent_id` 实现一级分类、二级分类树结构。
2. `edu_course.category_id` 指向课程所属分类，`edu_course_chapter.course_id` 指向课程主体。
3. `edu_course_evaluation.course_id` 指向课程，`edu_course_evaluation.user_id` 指向评价用户。
4. `user_study_record` 以 `user_id + course_id` 维护断点续学记录，`chapter_id` 指向最近学习章节。
5. `cms_home_section_item.target_id` 可指向课程或分类，用于首页推荐配置。

### 5.3 园地与互动关系

1. `garden_content.source_id` 指向内容来源表 `garden_content_source`。
2. `garden_content_comment.content_id` 和 `garden_content_like.content_id` 指向内容主体。
3. `user_favorite` 采用通用收藏模型，可扩展课程和园地内容。
4. `user_follow_source` 用于记录对内容来源的关注关系。
5. `risk_report_record` 可指向内容、评论或用户，用于举报处理和风控治理。

### 5.4 会员、成长与任务关系

1. `user_member_record` 存当前会员状态，`member_renewal_record` 存历史变更记录，`member_plan` 存套餐配置。
2. `user_growth_account` 存用户当前成长画像，`user_check_in_record` 存每日签到明细。
3. `growth_task_rule` 存任务规则，`user_task_record` 记录用户任务完成和奖励发放状态。

### 5.5 搜索、练习与运营关系

1. `user_search_history.user_id` 指向用户，`op_search_keyword` 用于维护热搜词、联想词和同义词配置。
2. `cms_recommend_slot` 与课程、分类、内容等资源通过 `resource_type + target_id` 建立弱关联。
3. `practice_question_bank` 与课程、章节为可选关联，`practice_question.bank_id` 指向题库。
4. `practice_paper` 与题库、课程、章节可建立关联，`practice_paper_question` 维护试卷与题目多对多关系。
5. `user_practice_record` 为用户答卷主表，`user_practice_answer` 为逐题作答明细，`user_wrong_question` 为错题沉淀表。

### 5.6 消息、客服与治理关系

1. `sys_message` 为消息主体，`sys_message_receiver` 为用户送达和已读明细。
2. `sys_message_template` 为模板配置表，可被 `sys_message.template_id` 复用。
3. `service_ticket` 为客服会话主表，`service_message` 为会话消息明细表，`service_faq` 为知识库条目。
4. `sys_sensitive_word` 与 `risk_report_record` 共同支撑风控审核和治理策略。

## 6. 索引与性能建议

1. 列表页高频查询字段必须建立普通索引，如状态、时间、分类、用户 ID、内容类型、试卷类型、举报状态。
2. 用户名、邮箱、手机号、角色编码、权限编码、功能开关编码、推荐位编码、模板编码建议建立唯一索引。
3. 评论、消息、客服消息、答题明细等明细表建议按主业务 ID + 时间建立联合索引。
4. 日志表、消息接收表、练习答题明细表建议预估数据量后考虑分表或按时间归档策略。
5. 园地内容的点赞数、评论数、收藏数，课程的评分数、评价数、收藏数建议做冗余字段，新增互动时同步更新计数。
6. 搜索历史、热搜词点击量、推荐位点击量等行为型数据可先存业务表，数据量提升后再引入埋点明细表或数仓。

3. 扫一扫功能业务尚未明确，当前仅在系统配置中预留开关，若涉及扫码内容、二维码资源或扫码登录，需要新增专门业务表。
4. 园地内容图片和客服附件当前建议使用 JSON 字段存储，若后续媒体资源更复杂，建议拆分统一媒体资源表。
5. 收藏当前采用通用表 `user_favorite`，若后续课程收藏、内容收藏、专题收藏差异变大，可以拆分为多个收藏表。
6. 客服模块当前按工单/会话模式设计，若后续采用即时通讯长连接方案，消息表和会话表需要进一步调整。
7. 练习中心当前按客观题自动判分设计，若后续扩展主观题、人工批改、防作弊，需要补充批改记录和考试控制表。
8. 统计表当前按日汇总设计，若系统数据量较小，也可以先不落地统计表，直接从业务表实时聚合。
