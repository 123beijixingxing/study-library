# 学习通前后端枚举常量清单

## 1. 文档说明

- 项目名称：学习通管理项目
- 文档版本：V1.1
- 编写日期：2026-04-24
- 对应文档：`docs/需求设计文档/学习通状态枚举字典与字段约束说明.md`
- 对应文档：`docs/需求设计文档/学习通接口清单.md`
- 对应文档：`docs/需求设计文档/学习通数据库表设计初稿.md`
- 文档目的：将系统中需要前后端统一的状态枚举、类型枚举和布尔标识沉淀为常量清单，便于后端建模、前端展示、接口联调和测试用例统一命名。

## 2. 统一约定

### 2.1 后端约定

| 项 | 建议 |
| --- | --- |
| 枚举类命名 | 使用 `PascalCase + Enum`，如 `UserStatusEnum` |
| 枚举项命名 | 使用 `UPPER_SNAKE_CASE`，如 `ENABLED` |
| 枚举值字段 | 使用 `value` 存接口/数据库值，使用 `label` 存中文展示文案 |
| 推荐方法 | `getValue()`、`getLabel()`、`fromValue()` |
| 推荐目录 | `backend/.../common/enums/` 或按业务域拆分 `.../enums/account/`、`.../enums/course/` |

### 2.2 前端约定

| 项 | 建议 |
| --- | --- |
| 常量对象命名 | 使用 `UPPER_SNAKE_CASE`，如 `USER_STATUS` |
| 类型命名 | 使用 `PascalCase`，如 `UserStatus` |
| 文案映射命名 | 使用 `XXX_LABEL`，如 `USER_STATUS_LABEL` |
| 推荐目录 | `src/constants/enums/`，按业务域拆分文件 |
| 值来源 | 与接口和数据库保持完全一致，不在前端自行改写值 |

### 2.3 接口与数据库约定

| 项 | 建议 |
| --- | --- |
| 存储值 | 数据库、接口、前端状态值统一使用英文小写或 `snake_case` |
| 展示值 | 页面中文文案统一从前端 `LABEL` 映射获取 |
| 布尔开关 | 优先统一 `0/1`，避免混用 `true/false` 和 `enabled/disabled` |
| 重复值 | 相同语义尽量复用公共枚举，不重复定义多套值域 |

## 3. 推荐实现模板

### 3.1 后端 Java 枚举模板

```java
@Getter
@AllArgsConstructor
public enum UserStatusEnum {
    ENABLED("enabled", "启用"),
    DISABLED("disabled", "禁用");

    private final String value;
    private final String label;

    public static UserStatusEnum fromValue(String value) {
        for (UserStatusEnum item : values()) {
            if (item.value.equals(value)) {
                return item;
            }
        }
        throw new IllegalArgumentException("Unknown user status: " + value);
    }
}
```

### 3.2 前端 TypeScript 常量模板

```ts
export const USER_STATUS = {
  ENABLED: 'enabled',
  DISABLED: 'disabled',
} as const

export type UserStatus = typeof USER_STATUS[keyof typeof USER_STATUS]

export const USER_STATUS_LABEL: Record<UserStatus, string> = {
  enabled: '启用',
  disabled: '禁用',
}
```

## 4. 公共可复用枚举

以下枚举建议作为前后端共享基础常量优先落地，避免多个业务域重复定义相同值域。

| 业务含义 | 后端枚举类 | 前端常量名 | 枚举值 | 适用位置 |
| --- | --- | --- | --- | --- |
| 启停状态 | `EnableStatusEnum` | `ENABLE_STATUS` | `enabled` / `disabled` | 用户、角色、权限、来源、套餐、任务规则、FAQ、热搜词、推荐位、模板、敏感词 |
| 展示状态 | `VisibleStatusEnum` | `VISIBLE_STATUS` | `visible` / `hidden` | 分类展示、课程评价、首页配置、园地内容展示 |
| 二值开关 | `BinaryFlagEnum` | `BINARY_FLAG` | `0` / `1` | 试看开关、会员专享、功能开关 |
| 审核状态 | `AuditStatusEnum` | `AUDIT_STATUS` | `pending` / `approved` / `rejected` | 园地内容审核 |

## 7. 落地优先级建议

| 优先级 | 建议先实现的常量 | 说明 |
| --- | --- | --- |
| P1 | `USER_STATUS`、`COURSE_STATUS`、`MESSAGE_TYPE`、`PRACTICE_STATUS`、`SERVICE_TICKET_STATUS`、`REPORT_STATUS` | 覆盖当前最核心业务闭环 |
| P2 | `MEMBER_STATUS`、`PAY_STATUS`、`QUESTION_TYPE`、`SEARCH_SCENE`、`RECOMMEND_RESOURCE_TYPE` | 支撑会员、练习、搜索运营能力 |
| P3 | `SENSITIVE_WORD_TYPE`、`SENSITIVE_WORD_MATCH_TYPE`、`MESSAGE_CHANNEL`、`RESULT_STATUS` | 用于风控和多渠道能力扩展 |

## 8. 待确认项

1. 后端若使用 Java，可直接采用枚举类；若使用 Go、Node.js 等技术栈，类名可不一致，但值域和常量名建议保持同一套语义。
2. 布尔型字段当前建议统一复用 `BinaryFlagEnum`，若前端更偏好 `true/false`，需要在 DTO 层做转换，数据库与接口值仍建议保留 `0/1`。
3. `EnableStatusEnum` 与部分业务域自定义状态是否完全复用，还是保留领域专有枚举类，需要结合后端代码风格确认。
4. `MessageChannelEnum` 当前包含短信、邮件、Push，若首版仅支持站内信，可先保留常量不启用渠道实现。
