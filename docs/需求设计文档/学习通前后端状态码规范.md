- 文档目的：统一前后端接口状态码、HTTP 状态语义、业务错误码分段规则和前端处理建议，作为接口实现、异常处理、联调和测试验收的统一依据。

## 2. 总体约定

### 2.1 返回结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "202604241030001234",
  "timestamp": 1776988200000
}
```

### 2.2 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | number | 业务状态码，成功固定为 `0` |
| `message` | string | 用户可读或联调用说明文案 |
| `data` | object | 成功时返回业务数据；失败时可为空 |
| `requestId` | string | 用于日志追踪和问题排查 |
| `timestamp` | number | 服务端响应时间戳 |

### 2.3 核心原则

1. HTTP 状态码描述传输层结果，业务状态码描述业务语义，两者必须同时清晰。
2. 成功响应统一 `code=0`，业务失败必须返回非 `0` 的明确错误码。
3. 同一业务问题在不同接口中应尽量复用同一个业务错误码，避免重复造码。
4. 状态前置条件不满足、资源不可访问、权限不足等场景必须优先返回可判定的专用错误码。
5. 面向用户的提示文案可以因场景定制，但 `code` 必须稳定，便于前端统一处理。

## 3. HTTP 状态码使用规范

| HTTP 状态码 | 适用场景 | 说明 |
| --- | --- | --- |
| `200` | 请求成功，或业务失败但系统成功返回标准响应体 | 推荐大多数业务接口使用 |
| `201` | 创建成功 | 可选，若项目统一用 `200` 也可接受 |
| `400` | 参数错误、枚举非法、格式错误 | 请求参数层面问题 |
| `401` | 未登录、Token 无效、Token 过期 | 鉴权失败 |
| `403` | 无权限、会员权限不足、资源禁止访问 | 权限或授权失败 |
| `404` | 资源不存在，或前台需隐藏内部状态时返回资源不可见 | 避免泄露内部状态 |
| `409` | 当前状态不允许该操作、重复提交、唯一性冲突 | 业务冲突类错误 |
| `429` | 发送验证码过频、接口频控、重复请求过快 | 限流类错误 |
| `500` | 服务内部异常 | 未预期错误 |
| `502/503` | 依赖服务异常、服务不可用 | 网关或外部依赖异常 |

## 4. 业务状态码分段规则

| 段位 | 范围 | 模块 |
| --- | --- | --- |
| 成功码 | `0` | 通用成功 |
| 通用错误 | `10000-10999` | 参数、幂等、上传、频控等 |
| 认证与登录 | `11000-11999` | 登录、验证码、Token |
| 用户与权限 | `12000-12999` | 用户、角色、权限 |
| 课程与分类 | `13000-13999` | 课程、章节、评价、分类 |
| 学习与练习 | `14000-14999` | 学习记录、试卷、错题、签到任务 |
| 园地与互动 | `15000-15999` | 园地内容、评论、收藏、举报提交 |
| 会员与成长 | `16000-16999` | 会员、套餐、支付、成长值 |
| 消息与客服 | `17000-17999` | 消息发送、已读、工单、FAQ |
| 搜索与运营 | `18000-18999` | 热搜词、推荐位、模板、配置 |
| 风控与审核 | `19000-19999` | 审核、举报处理、敏感词 |
| 系统错误 | `90000-90999` | 系统、数据库、远程服务、未知异常 |

## 8. 前端处理规范

### 8.1 通用处理建议

| 条件 | 前端建议 |
| --- | --- |
| `code = 0` | 正常处理数据 |
| `401` 或 `11002/11003` | 清空登录态并跳转登录页 |
| `403` 或 `16001/16002` | 显示权限提示；会员场景可跳转会员中心 |
| `404` | 显示资源不存在或内容已下线页 |
| `409` | 展示业务冲突提示，不建议自动重试 |
| `429` | 显示频控提示并限制按钮重复点击 |
| `500+` 或 `900xx` | 显示通用错误提示，并保留 `requestId` 便于排查 |

### 8.2 页面级建议

| 页面 | 常见错误码 | 建议处理 |
| --- | --- | --- |
| 登录页 | `11001`、`11009` | 显示登录失败原因 |
| 找回密码页 | `11004`、`11005`、`11006` | 验证码错误、过期或频控提示 |
| 课程详情页 | `13002` | 提示课程暂不可访问 |
| 练习中心 | `14003`、`14004`、`14005` | 提示会员限制或重复提交 |
| 我的消息页 | `17002` | 已读操作幂等处理即可 |
| 在线客服页 | `17006`、`17007` | 提示工单已关闭或状态不允许 |
| 园地内容页 | `15002`、`15003` | 提示内容已下线或不可见 |

## 9. 后端实现规范

1. 控制器层负责 HTTP 状态码和标准响应体封装，服务层负责抛出业务异常码。
2. 业务异常建议统一使用领域异常对象，至少包含 `code`、`message`、`httpStatus`。
3. 对 `STATE_NOT_ALLOWED`、`INVALID_ENUM_VALUE`、`MEMBER_REQUIRED` 这类高频错误建议封装公共异常构造方法。
4. 状态流转前置条件失败时，错误码优先使用模块专有码，通用码仅作为兜底。
5. 所有 `900xx` 错误必须记录服务端日志，并带上 `requestId`。

## 11. 待确认项

1. 当前建议大多数业务失败仍通过标准响应体返回，是否强制所有业务失败都使用 `200 + 非 0 code`，还是部分场景使用真实 `4xx/5xx`，需结合网关和前端拦截器策略确认。
2. 会员支付若接入真实支付平台，可能需要补充订单关闭、支付取消、退款中、退款失败等更细粒度状态码。
3. 若后续接入短信、邮件、Push 等多渠道消息，需要补充渠道级送达错误码和外部服务错误码细分。
