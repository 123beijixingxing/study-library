# 学习通后台真实接口切换与联调步骤

## 1. 文档说明

- 项目名称：学习通后台管理系统
- 对应前端工程：`study-library-admin`
- 文档目的：说明如何从本地 mock 演示模式切换到真实后端接口联调模式，并给出推荐联调顺序与问题排查方法。

## 2. 当前前端切换机制

`study-library-admin` 当前通过环境变量 `VITE_USE_MOCK` 决定请求走向：

- `true`：优先走 `src/mock/modules/*.ts`
- `false`：走真实接口地址

核心位置：

- 接口入口：`study-library-admin/src/api/modules/*.ts`
- 适配层：`study-library-admin/src/api/adapters/admin.ts`
- Base URL：`study-library-admin/src/api/utils/apiConfig.ts`

## 3. 切换前准备

在联调前请确认以下事项：

1. 后端已提供可访问的 admin 服务地址
2. 登录、刷新 token、退出登录接口已可用
3. 后端返回时间字段格式已明确
4. 后端枚举值已与接口文档一致，或已确认需要由 adapter 层适配
5. 前端联调人员可以访问浏览器控制台与 Network 面板

## 4. 环境切换方法

### 4.1 推荐做法

复制 `study-library-admin/.env.integration.example`，生成你本地真实联调配置，例如：

```bash
copy .env.integration.example .env.development.local
```

然后按实际后端地址修改。当前更推荐让浏览器始终请求前端同源地址，再由 Vite 代理转发到真实后端，这样局域网访问时不会触发浏览器跨域限制：

```env
VITE_API_BASE_URL=/api/admin/v1
VITE_UPLOAD_BASE_URL=/api/file/v1
VITE_USE_MOCK=false
VITE_DEV_PROXY_TARGET=http://127.0.0.1:8080
```

说明：

- `VITE_API_BASE_URL` / `VITE_UPLOAD_BASE_URL` 保持相对路径，浏览器只访问前端域名
- `VITE_DEV_PROXY_TARGET` 由 Vite 在开发期转发到真实后端
- 若你需要直接从浏览器访问后端地址，当前 dev 后端也已放开 `localhost`、`127.0.0.1` 和常见局域网网段的 CORS

### 4.2 最低要求

至少要保证：

- `VITE_USE_MOCK=false`
- `VITE_API_BASE_URL` 能正确落到真实 admin 接口前缀
- 若采用代理模式，`VITE_DEV_PROXY_TARGET` 必须指向真实后端地址

## 5. 联调命令

在 `study-library-admin` 下执行：

```bash
yarn
yarn dev
```

联调前建议先跑一次：

```bash
yarn type-check
```

联调完成后建议跑：

```bash
yarn build
```

## 6. 推荐联调顺序

### 6.1 第一阶段：登录与基础壳层

优先联调：

- `POST /api/admin/v1/auth/login`
- `POST /api/admin/v1/auth/refresh`
- `POST /api/admin/v1/auth/logout`
- `GET /api/admin/v1/auth/me`

验证点：

- 登录成功后 token、refreshToken 能写入 sessionStorage
- 401 时能自动刷新 token
- 刷新失败时能回到登录页
- 用户信息能正确展示在头部区域

### 6.2 第二阶段：主数据 CRUD

优先模块：

1. 用户管理
2. 课程管理
3. 题库管理
4. 练习试卷管理

验证点：

- 列表查询
- 新增保存
- 编辑保存
- 状态切换
- 删除动作
- 详情页回显一致

### 6.3 第三阶段：审核运营模块

优先模块：

1. 园地管理
2. 火热内容审核
3. 分享内容审核
4. 举报审核

验证点：

- 审核状态切换后页面立即正确回显
- 举报处理结果和备注字段能保存
- 前后台状态映射无偏差

### 6.4 第四阶段：会员 / 消息 / 客服 / 系统

优先模块：

1. 会员管理
2. 会员套餐配置
3. 成长规则
4. 消息管理 / 模板管理
5. 客服管理
6. 系统管理

### 6.5 第五阶段：分析与审计

优先模块：

1. 统计页
2. 日志页
3. 题目版本审计总台
4. 试卷分析资产库

## 7. 前端适配原则

当前前端已采用 adapter 分层，联调时请遵循：

1. 后端优先按接口文档提供字段
2. 若字段与页面结构不同，在 `src/api/adapters/admin.ts` 中适配
3. 不要在页面组件里零散写字段转换

常见适配示例：

- `userId -> id`
- `courseId -> id`
- `bankId -> id`
- `configId -> id`
- `createdAt / updatedAt -> createTime / updateTime / registerTime`

## 8. 常见联调问题排查

### 8.1 页面仍走 mock

排查项：

- `VITE_USE_MOCK` 是否真为 `false`
- 重启 dev server 了吗
- 浏览器缓存是否刷新

### 8.2 登录成功但自动退回登录页

排查项：

- token 是否写入 sessionStorage
- `GET /auth/me` 是否返回 401
- refreshToken 是否返回了有效刷新结果

### 8.3 列表接口有数据但页面为空

优先检查：

- 返回是 `data.list` 还是直接数组
- 分页字段是否是 `pageNum/pageSize/total/list`
- adapter 是否已经对目标字段做映射

### 8.4 日期时间显示为空

优先检查：

- 返回字段名是否在 adapter 中覆盖
- 时间是否为 ISO 字符串、时间戳或空值
- `resolveTimeText(...)` 是否已包含对应字段

### 8.5 状态枚举显示不正确

优先检查：

- 后端状态值是否与 `src/constants/enums.ts` 一致
- 若不一致，先在 adapter 层转换，再决定是否扩枚举

## 9. 联调完成后的最小验收动作

建议至少完成：

1. 登录 / 刷新 token / 退出登录
2. 用户管理一条完整 CRUD
3. 课程管理一条完整 CRUD
4. 题库管理 -> 题目管理 -> 题目详情完整链路
5. 练习试卷 -> 试卷分析页完整链路
6. 园地内容审核与举报处理完整链路
7. 会员 / 消息 / 系统至少各验证一个保存动作
8. `yarn build` 通过

## 10. 联调建议结论

当前前端已经具备：

- mock / 真实接口双模式切换能力
- token 与 refreshToken 流程骨架
- adapter 字段适配层
- 完整后台页面结构

因此下一阶段最合理的工作方式是：

1. 先切真实登录
2. 再按主数据 -> 审核运营 -> 配置消息 -> 分析审计顺序联调
3. 每完成一个模块就做一次页面验收和 build 验证
