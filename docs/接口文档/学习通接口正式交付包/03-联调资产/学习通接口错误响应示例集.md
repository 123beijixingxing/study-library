# 学习通接口错误响应示例集

## 1. 文档说明

- 项目名称：学习通管理项目
- 文档版本：V1.1
- 编写日期：2026-04-24
- 对应文档：`docs/需求设计文档/学习通前后端状态码规范.md`

- 对应文档：`docs/需求设计文档/学习通状态流转图说明.md`
- 文档目的：补充关键错误场景的标准响应报文示例，帮助前后端联调、测试验收和异常处理实现时保持一致。

## 2. 使用约定

| 项 | 说明 |
| --- | --- |
| 报文结构 | 统一使用 `code`、`message`、`data`、`requestId`、`timestamp` |
| 示例作用 | 展示推荐响应格式，不要求字段值完全固定 |
| 错误数据体 | `data` 可为空，也可返回当前状态、资源 ID、辅助提示等上下文 |
| HTTP 状态 | 示例同时给出推荐 HTTP 状态和业务错误码 |
| 前端处理 | 页面层应优先根据 `code` 处理业务逻辑，再根据 HTTP 状态做通用兜底 |

## 3. 通用错误响应骨架

```json
{
  "code": 10001,
  "message": "参数不合法",
  "data": null,
  "requestId": "202604241200001234",
  "timestamp": 1776993600000
}
```

## 5. 场景与错误码映射建议

| 场景类型 | 推荐业务码 | 建议是否返回上下文数据 |
| --- | --- | --- |
| 参数格式错误 | `10001` | 建议返回具体字段名 |
| 状态前置条件不满足 | `10007` 或模块专有码 | 建议返回 `currentStatus` |
| 会员权限不足 | `16001` / `16002` | 建议返回 `memberStatus` |
| 重复提交或重复处理 | `10005` 或模块专有码 | 建议返回业务记录 ID |
| 资源不可见 | `13002`、`15002` | 建议谨慎返回上下文，避免泄露过多内部信息 |

## 6. 联调使用建议

1. 联调时前后端先按本文示例对齐 `code`、`message`、`data` 结构，再细化具体字段。
2. 对于状态相关错误，建议优先返回 `currentStatus`、`memberStatus`、`auditStatus` 这类上下文字段，方便页面决定下一步动作。
3. 对于重复操作场景，若产品希望前端静默处理，可保持业务码不变，但允许前端忽略提示。
